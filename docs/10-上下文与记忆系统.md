## 1. 先分清：四本“账”

很多人把“上下文”和“记忆”混成一锅。代码里至少有四本账：

```text
A. conversation.history          真对话账本（进程内，同 session 累积）
B. api_conv                      当次发给模型的副本（可裁剪，用完可丢）
C. memories.md / MOZILCODE.md    跨会话持久知识（磁盘）
D. event_logs / events.jsonl     前端展示账（Daemon），默认不喂模型
```

| 账本 | 谁写 | 谁读 | 跨会话？ | 跨 Daemon 重启？ |
|------|------|------|----------|------------------|
| A history | Agent 主循环 / Daemon runner | 下一轮 LLM（经 B） | 同 session 进程内 | **默认否** |
| B api_conv | `prepare_api_conversation` | `client.stream` | 否 | 否 |
| C 记忆/指令文件 | 人写 or LLM extract | `load` → 注入 A | **是** | **是** |
| D UI 事件 | Daemon serialize | GUI WebSocket | 是（落盘） | 是（回放 UI） |

**学习口诀：**

- 多轮对话靠 **A**  
- 窗口压力靠 **B（便宜裁剪）** 和 **A 的 Layer2 有损压缩**  
- 跨聊天记住偏好靠 **C**  
- 屏幕上看到什么靠 **D**

## 2. 模型一次调用到底吃什么

```text
client.stream(
  api_conv,          # messages：来自 history 的发送视图
  system=system,     # 系统提示：角色规则等，通常不进 history
  tools=tools,       # 当前工具 JSON Schema
)
```

`api_conv.messages` 背后的 **真 history 典型结构**：

```text
[0]  环境上下文 user 消息
     - cwd / OS / 时间
     - agent catalog / skill catalog
     - 已激活 skill 正文
     （inject_environment，只插一次）

[1]  长期记忆 user 消息（包在 <system-reminder>）
     - # mozilcodeMd   ← MOZILCODE.md 指令
     - # autoMemory    ← memories.md
     - # currentDate
     （inject_long_term_memory，只插一次）

[2..] 多轮对话
     user / assistant(+tool_use/thinking) / user(+tool_results) ...

末尾可能还有本轮临时：
     - Team 通知
     - Plan 模式 reminder
     - Hook notification
     - deferred 工具名单 reminder
```

另外：

- `system` 由 `build_system_prompt(...)` 单独拼，可含 Hook 的 prompt 注入  
- `tools` 来自 `ToolRegistry.get_all_schemas`（不含未发现的 deferred）

## 3. 会话上下文：从空 history 到可续聊

### 3.1 谁持有 conversation

| 入口 | 行为 |
|------|------|
| Daemon | 每 `session_id` 一个 `ConversationManager`，多轮 `add_user_message` 追加 |
| TUI | 进程内持有 |
| 子 Agent | 通常新 conversation 或裁剪任务上下文 |

Daemon 侧（复习）：

```text
POST /api/task
  → conversation.add_user_message(prompt)
  → agent.run(conversation)   # 不新建 history
```

### 3.2 `run()` 开头：组装工作集

`agent/core.py` 逻辑顺序：

```text
1. env_context = build_environment_context(work_dir, skills, catalogs)
2. memory_content = memory_bridge.load_context(最近 user query)
3. inject_agent_context(conversation, env, instructions, memory_content)
4. session_start hooks
5. while iteration:
     ...
```

`inject_agent_context`（`agent/compaction.py`）只是两行：

```text
conversation.inject_environment(environment_context)
conversation.inject_long_term_memory(instructions_content, memory_content)
```

### 3.3 注入为什么“只做一次”

`ConversationManager` 有两个标志：

| 标志 | 作用 |
|------|------|
| `env_injected` | 防止环境块重复 insert |
| `ltm_injected` | 防止 LTM 块重复 insert |

`replace_history`（Layer2 压缩后）会 **清零这两个标志**，于是必须 `reinject_after_compact` 再插一遍。

### 3.4 token 怎么估（决定何时压缩）

```text
record_usage_anchor(input, output, cache_*)
  → baseline_tokens = 真实 API 计费相关合计
  → anchor_count = 当时 history 长度

current_tokens()
  有锚点: baseline + estimate(history[anchor_count:])
  无锚点: estimate(整个 history)   # 冷启动或刚 compact
```

字符启发式：约 `chars / 3.5`（见 `conversation.py`）。

## 4. Layer1：tool-result 预算（可逆、偏发送侧）

### 4.1 目的

工具输出（尤其 `ReadFile`、bash）极易把 prompt 撑爆。  
Layer1 在 **每次调用 LLM 前** 生成发送副本，**默认不改真 history 语义**（设计 B：复制 messages）。

入口：

```text
prepare_api_conversation(conversation, session_dir, replacement_state)
  → apply_tool_result_budget(...)
  → 可能 append_replacement_records 落盘决策记录
```

### 4.2 常量（`context/tool_results.py`）

| 常量 | 值 | 含义 |
|------|-----|------|
| `SINGLE_RESULT_CHAR_LIMIT` | 50_000 | 单条超限 → 落盘 + 预览 |
| `INLINE_RESULT_CHAR_LIMIT` | 10_000 | 单条偏大 → 截断 |
| `AGGREGATE_CHAR_LIMIT` | 200_000 | 全 tool_result 合计超限 → 从大到小继续落盘 |
| `PREVIEW_CHARS` | 2_000 | 落盘预览长度 |
| `KEEP_RECENT_TURNS` | 10 | 最近 N 个“完成 turn”的结果尽量保留 |
| `OLD_RESULT_SNIP_CHARS` | 2_000 | 更早结果再 snip 的阈值相关 |

落盘目录：

```text
{work_dir}/.mozilcode/session/tool-results/{tool_use_id}.txt
```

预览形态带 `<persisted-output>` 标签，告诉模型：全文在文件路径，这里只有前 2KB。

### 4.3 三遍处理（直觉版）

```text
对 history 中每条带 tool_results 的消息：

Pass 0  已有 replacement_state 决策 → 直接复用
Pass 1  fresh 且单条 > 50k → persist + preview，记入 state
Pass 2  剩余合计 > 200k → 按长度从大到小继续 persist
Pass 3  对“够老”的 turn 的 tool_result → <snipped> 短预览

返回 new ConversationManager（拷贝了 env/ltm/token 锚点字段）
```

`ContentReplacementState`：

- `replacements[id] = 替换后文本`  
- `seen_ids`：已见过、本轮不再当 fresh 处理  

### 4.4 Layer1 不会做什么

- 不会删用户原文消息  
- 不会写 `memories.md`  
- 不保证 UI 事件变短（UI 用的是 Agent 事件流，不是 api_conv）

## 5. Layer2：自动压缩（有损、改真 history）

### 5.1 何时触发

每轮 iteration 开头：

```text
current = conversation.current_tokens()
threshold = context_window - SUMMARY_OUTPUT_RESERVE(20000) - AUTO_COMPACT_SAFETY_MARGIN(13000)
if current >= threshold: auto_compact(...)
```

手动压缩（Daemon `POST /api/compact/{sid}` / Agent API）走同一套，但 margin 更小（`MANUAL_COMPACT_SAFETY_MARGIN=3000`），且可忽略“未超阈值”。

### 5.2 保留窗口怎么定

从 **尾部向前** 累加 token：

| 参数 | 值 | 含义 |
|------|-----|------|
| `KEEP_RECENT_TOKENS` | 10_000 | 目标：至少保留约这么多近期原文 token |
| `MIN_KEEP_MESSAGES` | 5 | 或至少 5 条消息 |
| `KEEP_MAX_TOKENS` | 40_000 | 硬上限，防止一条超大消息吞掉全窗口 |

再 `_align_keep_start_to_tool_pair`：避免 `tool_use` / `tool_result` 被拦腰拆开。

前缀太小（`< MIN_SUMMARIZE_PREFIX_TOKENS` 约 2000）→ **不压**，避免“压了个寂寞”。

### 5.3 摘要怎么做

```text
to_summarize = history[:keep_start]
keep_tail    = history[keep_start:]

另建 summary_conv，塞 SUMMARY_PROMPT + 前缀消息
client.stream(...) 收集文本
extract_summary：优先取 <summary>...</summary> 内文
```

失败：

- 最多重试；prompt 过长时丢掉部分旧 turn 再试  
- `CompactCircuitBreaker` 连续失败 3 次熔断自动压缩  

### 5.4 重建 history

```text
attachment = build_recovery_attachment(recovery, tool_schemas)
  # 最近读过的文件、技能 SOP 等快照（有条数/token 预算）

new_messages =
  [ 一条 user：说明“会话被压缩”+ 摘要 + 可选 transcript 路径 + attachment ]
  + keep_tail

conversation.replace_history(new_messages)
cleanup_tool_results(session_dir)   # 清 Layer1 落盘缓存
reinject_after_compact(...)         # 重新 load 记忆 + 注入 env/指令/记忆
```

压缩后 UI 会收到 `CompactStarted` / `CompactNotification` / 更新后的 `UsageEvent`。

### 5.5 RecoveryState 是什么

Layer2 摘要会丢掉“刚读的文件正文”。`RecoveryState` 在工具执行路径上记录：

- 最近文件读（默认附件最多 5 个文件级快照，带 per-file token 预算）  
- 技能调用正文  

压缩后拼进摘要消息，减少“失忆式工作集清空”。

## 6. 记忆系统：跨会话的持久层

### 6.1 两类“长期内容”

| 类型 | 来源 | 路径示例 | 更新方式 |
|------|------|----------|----------|
| **项目/用户指令** | 人写 | `MOZILCODE.md`、`.mozilcode/MOZILCODE.md`、`~/.mozilcode/MOZILCODE.md` | 手改；`@include` 展开 |
| **自动记忆** | LLM 提取 | `~/.mozilcode/memories.md`、`<project>/.mozilcode/memories.md` | `MemoryManager.extract` |

`load_instructions(project_root)` 按固定顺序读上述 MOZILCODE 路径，用 `---` 拼接；`@include` 不得逃出项目根。

自动记忆四个分类：

| 分类 | 写入层级 |
|------|----------|
| 用户偏好 | 用户级 `~/.mozilcode/memories.md` |
| 纠正反馈 | 用户级 |
| 项目知识 | 项目级 `<project>/.mozilcode/memories.md` |
| 参考资料 | 项目级 |

### 6.2 模块分层

```text
配置 MemoryConfig
    │
    ▼
build_memory_hub(config, work_dir)
    │  默认 builtin.markdown → MarkdownMemoryProvider
    ▼
MemoryHub(providers=[...])
    │  load_context / observe / search / write
    │  单 Provider 失败隔离；load 超时 3s / observe 10s
    ▼
AgentMemoryBridge(memory_hub, client, protocol, project_root)
    │  load_context / observe / extract_memories
    ▼
Agent.run 在固定时机调用
```

`memory.enabled = false` → `build_memory_hub` 返回 `None` → Bridge 全部空操作。

### 6.3 三个时点（务必背）

#### 时点 A：读（`run()` 开头 + compact 后 reinject）

```text
query = latest_user_query(conversation)   # 最近真实用户话
memory_content = hub.load_context(query, scope)

MarkdownMemoryProvider.load_context:
  return MemoryManager.load()
  # 用户级 + 项目级 memories.md 原文拼接
  # 注意：当前内置实现基本是“全量加载”，不是向量检索 top-k
```

然后进入 LTM 注入（见 §3）。

#### 时点 B：轻观察（本轮结束且无 tool_calls）

```text
asyncio.ensure_future(observe(MEMORY_EVENT_TURN_COMPLETED))
```

- 不阻塞主循环  
- 内置 Markdown Provider **忽略** `turn_completed`（只认 `turn_committed`）  
- 预留给其他 Provider 做轻量统计  

#### 时点 C：重提取（每 5 次“完成轮”）

```text
MEMORY_EXTRACTION_INTERVAL = 5
_loop_count % 5 == 0
  → extract_memories()
  → observe(MEMORY_EVENT_TURN_COMMITTED)
  → MarkdownMemoryProvider → manager.extract(client, conversation, protocol)
```

`extract` 算法：

```text
1. current = load() 两边 memories.md
2. recent = history[last_extraction_msg_count:]   # 增量
3. 只拼有文本的 user/assistant 行（忽略纯 tool 壳）
4. 构造独立 ConversationManager + 提取 system
5. client.stream 收集完整 markdown
6. _write_memories：
     按 ### 标题分节
     过滤 "- " 条目与占位符
     用户偏好/纠正反馈 → 用户文件（整文件覆写该层内容）
     项目知识/参考资料 → 项目文件
7. 更新 last_extraction_msg_count
```

约束：

- `_extracting` 防重入  
- 异常吞掉，只打 debug  
- **另开 LLM 调用**，不污染当前会话 history  

### 6.4 记忆进模型的“形状”

`inject_long_term_memory` 会把指令和记忆包进：

```text
<system-reminder>
As you answer the user's questions, you can use the following context:
# mozilcodeMd
...
# autoMemory
...
# currentDate
...
IMPORTANT: this context may or may not be relevant ...
</system-reminder>
```

即：对 API 来说它是一条 **user 消息**，语义上是系统提醒，不是 assistant 独白。

## 7. 完整时间轴（建议对着 core.py 读）

```text
用户消息进入 conversation.history
                │
                ▼
┌────────── Agent.run 开始 ──────────┐
│ load 记忆 + inject env + inject LTM │
│ session_start hooks                 │
└────────────────┬───────────────────┘
                 ▼
        ┌── iteration 循环 ──┐
        │ turn_start         │
        │ 外部通知注入        │
        │ Layer2 阈值检查     │──超限──► 摘要+keep_tail
        │                    │         replace_history
        │                    │         reinject 记忆/环境
        │ pre_send           │
        │ system + reminders │
        │ Layer1 → api_conv  │
        │ stream LLM         │
        │ 写回 history       │
        │ 有 tool？          │
        │  是 → 执行工具     │──► tool_results 进 history
        │       （可记 recovery 文件/技能）
        │       continue     │
        │  否 → 结束         │
        │     observe 异步   │
        │     每5轮 extract  │──► 写 memories.md
        │     LoopComplete   │
        └────────────────────┘
```

第二轮用户消息：

```text
同一 ConversationManager
history 仍含：env、LTM、上一轮工具轨迹（除非被 Layer2 摘要掉）
append U2 → 再 run
  若 env/ltm 已 injected 且未 compact → 不会重复插 LTM
  新 extract 写入磁盘后，要等下次 inject/reinject 才进模型视野
```

## 8. 状态机与标志位

### 8.1 ConversationManager

| 字段 | 含义 |
|------|------|
| `history` | 消息列表 |
| `env_injected` / `ltm_injected` | 注入闸门 |
| `baseline_tokens` / `anchor_count` | API 用量锚点 |
| `last_input_tokens` | 兼容旧字段，与 baseline 同步 |

### 8.2 Agent 侧记忆

| 字段 | 含义 |
|------|------|
| `memory_hub` | 可插拔中心，可 None |
| `memory_bridge` | 统一 load/observe/extract |
| `_loop_count` | 完成轮计数，驱动提取间隔 |
| `instructions_content` | 创建时读入的 MOZILCODE 文本 |
| `replacement_state` | Layer1 决策缓存 |
| `recovery_state` | Layer2 恢复附件源 |
| `compact_breaker` | 自动压缩熔断 |
| `session_dir` | tool-results 等会话目录 |

### 8.3 事件名

| 常量 | 值 | 内置 Markdown 是否处理 |
|------|-----|------------------------|
| `MEMORY_EVENT_TURN_COMPLETED` | `turn_completed` | 否 |
| `MEMORY_EVENT_TURN_COMMITTED` | `turn_committed` | **是 → extract** |
| `MEMORY_EVENT_SESSION_END` | `session_end` | 否（可扩展） |

## 9. 与 Daemon / UI 的边界

```text
Daemon 负责：
  持有 conversation（A）
  序列化 AgentEvent → event_logs（D）
  手动 compact API 调到同一套 Layer2

Daemon 不负责：
  把 events.jsonl 还原成 history（当前主路径没有）
  在 HTTP 响应里塞完整上下文

因此：
  重启 Daemon 后
    C 记忆/指令还在 → 新 session 仍可 load
    A 多轮细节丢了 → 模型像“新聊天但还记得偏好”
    D UI 仍可回放旧气泡 → 别误以为模型也记得全文
```

## 10. 排障表

| 现象 | 可能原因 | 先查 |
|------|----------|------|
| 模型不遵守项目规范 | MOZILCODE.md 路径不对 / 未 load | `load_instructions`、`instructions_content` |
| 改了 memories.md 当前对话无感 | 已 `ltm_injected`，未 compact/reinject | 标志位；是否新 run |
| 总是过早压缩 | `context_window` 配小了或估大了 | Provider 配置、`current_tokens` |
| 压缩后不会干活 | 摘要丢细节且 recovery 空 | 是否读过文件进 RecoveryState |
| 工具输出模型只见预览 | Layer1 落盘/截断 | `tool-results/`、`replacement_state` |
| 记忆从不增长 | hub 禁用 / 未满 5 轮 / extract 失败 | `memory.enabled`、`_loop_count`、日志 |
| 用户偏好写到项目文件 | 标题分类关键词不匹配 | extract 输出是否 `### 用户偏好` |
| UI 有历史模型失忆 | 重启后只有 D 没有 A | 是否新 runtime 空 history |
