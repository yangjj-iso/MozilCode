## 1. 什么叫「驾驭工程」

本项目里的 Agent 能力很强：读文件、改代码、跑命令、拉 MCP、开子 Agent。  
**驾驭工程**不是某个单独模块名，而是一套贯穿装配与运行时的控制面：

```text
让模型“能干活”
    同时保证：
    1) 只能在允许的边界内干活
    2) 危险动作必须先过闸
    3) 人能随时确认 / 拒绝 / 取消
    4) 失败不会无限放大
    5) 扩展点可插拔，但不绕开核心闸门
```

可以记成五句：

| 原则 | 项目里的落点 |
|------|--------------|
| **圈地** | `work_dir` + `PathSandbox` |
| **分级放行** | `PermissionMode` + 规则 + 危险命令 |
| **人在回路** | `PermissionRequest` / `AskUserRequest` + Future |
| **可扩展但可拦截** | Hook（尤其 `pre_tool_use` 可 reject） |
| **有限资源** | `max_iterations`、compact 熔断、cancel 任务 |

## 2. 一张总图：控制面夹着能力面

```text
用户 / GUI / A2A
        │
        ▼
┌──────────── Daemon 控制面 ────────────┐
│ session / task / cancel / mode / WS   │
└────────────────┬──────────────────────┘
                 ▼
┌──────────── Agent 运行时 ─────────────┐
│ factory 装配：client/tools/perm/hook  │
│ run 循环：                             │
│   Hook 生命周期                        │
│   Layer2 压缩熔断                      │
│   LLM 调用                             │
│   工具：pre_hook → 权限 → 执行 → post │
└────────────────┬──────────────────────┘
                 ▼
        项目文件 / Shell / MCP / 子 Agent
```

**能力面**（LLM + Tools）负责“做什么”。  
**控制面**（权限/Hook/会话/取消/预算）负责“允不允许、何时停、谁确认”。

## 3. 装配时就先把缰绳系上

`create_agent_from_config(...)`（`agent/factory.py`）不是“只 new 一个 Agent”，而是**把控制依赖一次性焊死**：

```text
AppConfig + work_dir + permission_mode
        │
        ├─ create_client(provider)          # 模型通道
        ├─ load_instructions(work_dir)      # 项目指令（软驾驭：告诉它怎么做）
        ├─ build_memory_hub(...)            # 长期偏好（软驾驭）
        ├─ ToolRegistry + MCP 工具          # 能力清单
        ├─ PermissionChecker(               # 硬驾驭
        │     DangerousCommandDetector,
        │     PathSandbox(work_dir),
        │     RuleEngine(user/project/local),
        │     mode
        │  )
        ├─ HookEngine（可选，Daemon 全局可共享）
        └─ Agent(..., permission_checker, hook_engine, context_window, ...)
```

学习点：

1. **硬驾驭**（权限/沙箱）和 **软驾驭**（MOZILCODE.md / memories / system prompt）同时存在。  
2. 权限对象绑在 Agent 上，不靠模型自觉。  
3. MCP 失败只 warning，不阻断本地工具——外部扩展不能拖垮主控。

## 4. 六层硬闸门：工具真正执行前发生了什么

主循环里一次工具调用（串行路径简化）：

```text
LLM 产出 tool_call
   │
   ▼
① 工具是否存在 / 是否 enabled          ToolRegistry
   │
   ▼
② pre_tool_use Hook（可 reject）       hooks + agent/tool_hooks.py
   │
   ▼
③ PermissionChecker.check(...)         permissions/checker.py
   │  deny → 错误结果，不执行
   │  ask  → yield PermissionRequest，卡住等 Future
   │  allow → 继续
   ▼
④ tool.execute(...)
   │
   ▼
⑤ post_tool_use Hook（不可 reject）
   │
   ▼
⑥ tool_result 写回 conversation.history
```

并发批处理也是「先逐个 pre_hook + 权限预检，通过后再并行执行」。

### 4.1 工具存在与启用（能力闸）

`ToolRegistry`：

- `register`：装上能力  
- `disable` / `is_enabled`：模式或策略可关掉某工具  
- `should_defer` + `ToolSearch`：**先不把全部 schema 塞给模型**（降低误用与上下文膨胀）

这是“不给枪”和“枪先锁在柜子里”两种驾驭。

### 4.2 Hook：可插拔拦截

生命周期事件见 `hooks/events.py`，最关键的是：

| 事件 | 驾驭含义 |
|------|----------|
| `session_start` / `session_end` | 会话边界 |
| `turn_start` / `turn_end` | 每轮边界 |
| `pre_tool_use` | **可 reject，硬拦工具** |
| `post_tool_use` | 事后审计/格式化，不能撤销已执行 |
| `pre_send` / `post_receive` | 改模型输入输出视野 |
| `compact` / `error` | 系统态观察 |

配置在 `config.yaml` 的 hooks 列表，由 `load_hooks` 装入 `HookEngine`。

### 4.3 PermissionChecker：固定分层决策

`permissions/checker.py` 的顺序是代码真相（不要靠猜）：

```text
Layer 0  Plan 模式例外
         - 允许 Agent/ToolSearch/AskUser/ExitPlanMode
         - 允许写计划文件（.mozilcode/plans 下）

Layer 1  命令安全
         - 只读安全命令 → allow
         - 危险命令黑名单 → deny

Layer 2  路径沙箱（read/write）
         - 目标必须在 project_root 或额外允许根（含 temp）

Layer 3  规则引擎
         - user / project / local 三级 permissions.yaml
         - 命中 allow/deny 即返回

Layer 4  PermissionMode 兜底
         - 按工具 category: read|write|command

Layer 5  仍不确定 → ask（人在回路）
```

### 4.4 模式矩阵（策略开关）

`permissions/modes.py`：

| 模式 | read | write | command | 典型用途 |
|------|------|-------|---------|----------|
| `default` | allow | ask | ask | 日常 |
| `acceptEdits` | allow | allow | ask | 自动改代码，命令仍问 |
| `plan` | allow | ask* | ask | 先规划；计划文件有例外 |
| `bypassPermissions` | allow | allow | allow | 全放行（仍受沙箱/危险命令等更前层约束） |
| `custom` | ask | ask | ask | 几乎事事确认 |
| `dontAsk` | allow | allow | allow | 非交互/自动跑（配合 noninteractive 路径） |

注意：`bypass` / `dontAsk` **不是“关掉整个 checker”**。  
危险命令 deny、路径沙箱 deny 仍可能在更前层生效。

运行时切换：

- TUI：`app.py` / slash command  
- Daemon：`POST /api/session/{sid}/mode` → `agent.set_permission_mode`  
- `set_permission_mode` 同步改 `permission_checker.mode`

### 4.5 规则与“永远允许”

`RuleEngine` 三级文件：

```text
~/.mozilcode/permissions.yaml                 # 用户级
<project>/.mozilcode/permissions.yaml         # 项目级
<project>/.mozilcode/permissions.local.yaml   # 本机/会话追加
```

用户在弹窗选 **ALLOW_ALWAYS** 时：

```text
authorize_tool_call
  → 根据 content 生成 pattern
  → rule_engine.append_local_rule(allow)
```

这是“把一次人工确认沉淀成可复用策略”。

### 4.6 人在回路（HITL）如何阻塞主循环

```text
Agent 内：
  future = asyncio.Future()
  yield PermissionRequest(..., future=future)
  response = await future          # 这里卡住，不继续执行工具

Daemon 序列化：
  Future 不能 JSON
  → serialize_event 去掉 future，换成 request_id=str(id(future))
  → 写入 event_logs / WS 推前端

前端：
  用户点允许/拒绝
  → POST /api/permission/...（带 request_id）
  → Session 找到 future 并 set_result

Agent 被唤醒：
  ALLOW / DENY / ALLOW_ALWAYS → 继续或返回 Permission denied
```

`AskUserQuestion` 同构：也是 Future 握手，不是轮询。

非交互路径（`run_to_completion` / `noninteractive_tools`）：

- **不 yield** PermissionRequest  
- 按 dontAsk/自动策略处理  
- 仍走 pre_tool_use 与参数校验  

子 Agent 常走这条“无人值守”分支。

## 5. 圈地：项目边界与作用域

### 5.1 work_dir 是第一边界

Daemon 创建 session 时写入 `meta.work_dir`，再：

```text
create_agent_from_config(config, work_dir, permission_mode)
  → PathSandbox(work_dir)
  → tools base_dir = work_dir
  → memory/instructions 按 work_dir 加载
```

工具读写、记忆、指令、worktree 都围着这个根转。  
**不是从 prompt 猜项目目录。**

### 5.2 PathSandbox

允许根默认：

1. `project_root`（work_dir resolve）  
2. 系统 temp  
3. 可选 `extra_allowed`

路径 resolve 后必须 `relative_to` 某允许根，否则 deny。  
这是文件系统级缰绳，比模型“答应不乱写”可靠。

### 5.3 Worktree 是受控搬家，不是逃逸

进入 worktree：

- 更新 `agent.work_dir` + session meta  
- 工具与沙箱基准跟着变  
- 仍在会话控制与权限模式下  

详见 08 / worktree 模块。

## 6. 软驾驭：提示、指令、计划模式

硬闸门决定“能不能做”。软驾驭影响“想怎么做”：

| 机制 | 作用 |
|------|------|
| system prompt | 角色与通用工程规范 |
| `MOZILCODE.md` / instructions | 项目级 SOP |
| memories.md | 跨会话偏好 |
| Plan mode + ExitPlanMode | 先写计划文件，再退出计划模式动手 |
| Skill / Agent catalog | 引导选用特定工作流或子代理 |

Plan 模式在权限层有**硬约束配合软约束**：只放行特定工具 + 计划文件写入。

## 7. 资源与失败边界（防失控）

### 7.1 迭代上限

`Agent.max_iterations`（默认 50）：

```text
单次 run() 内 iteration 超限
  → 结束并给出 “reached maximum iterations”
```

防止工具来回调用死循环。

### 7.2 上下文预算（也是驾驭）

| 层 | 作用 |
|----|------|
| Layer1 tool-result budget | 单次发送体积可控 |
| Layer2 auto-compact | 窗口逼近时有损摘要，保住可续聊 |
| CompactCircuitBreaker | 连续失败 3 次熔断自动压缩，避免失败风暴 |

上下文管理详 [10](./10-上下文与记忆系统.md)。这里强调：  
**预算控制 = 防止模型被超长历史拖死，也防止压缩逻辑本身失控。**

### 7.3 任务取消

```text
前端 / WS action=cancel / POST /api/session/{sid}/cancel
  → ActiveTaskRegistry.cancel
  → asyncio.Task.cancel
  → 发出 Task cancelled 事件

关 session：
  cancel_pending futures + 停活动任务 + 清 runtime
```

人不但能“问一次再做”，也能“整轮停掉”。

### 7.4 配置与产品面边界

`removed_capabilities`：

- 配置加载拒绝已删除能力段  
- Daemon 路由注册 assert 不得暴露已移除 surface  

这是**产品演进的驾驭**：旧能力不能从配置或路由偷偷复活。

## 8. Daemon：把“驾驭”暴露成可操作 API

没有 Daemon 时，TUI 同进程直接 `Agent.run`。  
有 Daemon 时，控制面被协议化：

| 用户意图 | 协议 |
|----------|------|
| 开一条会话线程 | `POST /api/session` |
| 发一条用户消息并跑 Agent | `POST /api/task` |
| 改权限模式 | `POST /api/session/{sid}/mode` |
| 回答权限/提问 | permission / ask 回传 API |
| 取消当前前台任务 | `POST /api/session/{sid}/cancel` 或 WS cancel |
| 看进度 | `WS /ws/{sid}` 事件流 |

Session / Task 分层（复习）：

```text
Session = 长寿命线程（持有 Agent + history + 模式）
Task    = 一次用户发送触发的前台执行（可取消）
```

Daemon 不替代权限引擎；它负责：

1. 持有 runtime  
2. 把 Future 事件变成可网络回传的 request_id  
3. 把取消/模式切换变成 HTTP 动作

## 9. 端到端：一次“写文件”如何被驾驭

```text
用户：把 foo.py 里的 bar 改成 baz
  │
  ▼
Daemon start_task → conversation.add_user_message → agent.run
  │
  ▼
LLM 决定调用 EditFile
  │
  ▼
pre_tool_use hooks（策略脚本可 reject）
  │
  ▼
PermissionChecker
  - 路径是否在 work_dir 沙箱内？否 → deny
  - 规则是否 allow/deny？
  - mode=default 且 write → ask
  │
  ▼
yield PermissionRequest → GUI 弹窗
用户 ALLOW
  │
  ▼
EditFile.execute
  │
  ▼
post_tool_use（可跑 formatter/lint 通知）
  │
  ▼
tool_result 进 history → 可能再调 LLM → 无 tool_call → LoopComplete
```

若用户点 DENY：工具不执行，模型只看到 “Permission denied…”，会话仍在。

## 10. 软硬对照表（学习记忆用）

| 类型 | 机制 | 失败时会怎样 |
|------|------|--------------|
| 硬 | 沙箱 deny | 工具不执行 |
| 硬 | 危险命令 deny | 工具不执行 |
| 硬 | 规则 deny | 工具不执行 |
| 硬 | pre_tool_use reject | 工具不执行 |
| 硬 | 用户 DENY | 工具不执行 |
| 硬 | cancel task | 本轮 run 中断 |
| 硬 | max_iterations | 本轮 run 结束 |
| 半硬 | Layer1/2 上下文 | 模型视野变短/被摘要 |
| 软 | system/instructions/memory | 模型可能忽略（需靠硬闸兜底） |
| 软 | Skill 目录提示 | 不强制执行 |

**设计直觉：软驾驭提高“做对”的概率；硬驾驭保证“做错时出不了圈”。**

## 11. 模块职责边界（防复杂度扩散）

```text
permissions/     只做决策：allow/deny/ask + reason
agent/tool_authorization.py
                 把决策接到主循环：deny 结果 / ask Future / allow 继续
hooks/           可插拔副作用与拦截，不替代权限矩阵
tools/registry   能力清单与启用/延迟
daemon/          会话任务协议、序列化、取消；不实现具体工具
context/         预算与压缩；不负责权限
```

不要把“再加一种业务限制”直接塞进 LLM prompt 就完事；  
若必须保证，优先落在：

1. checker 分层  
2. registry disable  
3. pre_tool_use reject  
4. 工具自身参数校验

## 12. 排障

| 现象 | 先查 |
|------|------|
| 总是弹权限 | mode 是否 default/custom；规则是否未 allow；category 是否 write/command |
| 从不弹权限 | bypass/dontAsk；规则 allow；安全只读命令；checker 为 None |
| 路径被拒 | work_dir；是否 symlink 逃出；绝对路径是否在沙箱外 |
| 危险命令仍执行？ | 是否被 is_safe_command 误判；是否非 Bash/category 不是 command |
| Hook 拦不住 | 是否挂在 pre_tool_use；condition 是否匹配；引擎是否装上 |
| 前端点了允许无反应 | request_id 是否对上；session 是否还在；future 是否已 done |
| 子 Agent 乱改文件 | 子 Agent permission_mode；noninteractive 策略；沙箱 work_dir |
| 取消无效 | 取消的是前台 active task 还是 background task id |

## 13. 建议阅读顺序

1. `agent/factory.py`：缰绳如何在装配期系上  
2. `permissions/modes.py` + `checker.py`：决策顺序  
3. `agent/tool_authorization.py`：ask 如何变成 Future  
4. `daemon/serialize.py` + permission 回传路由：跨进程人机握手  
5. `hooks/events.py` + `agent/tool_hooks.py`：可插拔拦截点  
6. `agent/core.py` 工具执行段：控制点在主循环的真实顺序  
7. `tools/registry.py`：启用/延迟暴露  
8. `context/manager.py` 的 `CompactCircuitBreaker`：失败熔断  

---

## 14. 一句话

**驾驭工程 = 用 work_dir/沙箱圈地，用权限分层与规则分级放行，用 Hook 插拔拦截，用 Future 做人在回路，用 Daemon 暴露取消与模式切换，用迭代/上下文/熔断限制失控；**  
模型负责提议动作，系统负责决定哪些动作真正落地。
